Introduction

The toys in ToyLand yearn to be played with. The more they are played with, the happier they become, until one day they reach their maximum happiness level and retire off to toy heaven. In this lab you will develop a Java program that leverages inheritance to help the toys achieve their hopes and dreams.

Before working on this assignment, you must have completed the problem solving work!

Design

As you do the work described in the rest of this document, you must follow the supplied public design for this lab. This class diagram details all the classes and their relationships.

IToy

IToy is an interface for the public behaviors of Toy. It is used by the Junit tests, and the main program.

Toy

Toy is the abstract superclass of the toy inheritance hierarchy. All toys must inherit from it to obtain the following behaviors:

  • A constructor that takes the product code and name.
  • getProductCode(): Returns the unique integer product code of the toy.
  • getName(): Returns the name of the toy.
  • getHappiness(): Returns the toys happiness (starts at INITIAL_HAPPINESS and increases when played with).
  • isRetired(): A toy is retired when its happiness level reaches/exceeds MAX_HAPPINESS (starts as not retired).
  • getWear(): Returns the toys current wear amount (starts at INITIAL_WEAR and increases when played with).
  • increaseWear() Increase the wear by the amount passed in.
  • play(): When played with, the following things happen in order:
    1. A playing message is displayed to standard output in the format (## is the number of minutes): 
                      PLAYING(##): toy-toString
      For example: 
                      PLAYING(40): Toy{PC:100, N:Play-Doh, H:0, R:false, W:0.0}, PlayDough{C:GREEN}
    2. The special play is invoked for the same number of minutes.
    3. The happiness of the toy increases by the number of minutes.
    4. If the happiness of the toy reaches or exceeds MAX_HAPPINESS, the toy retires and displays a message to standard output in the format: 
      5.                 RETIRED: toy-toString
      For example: 
                      RETIRED: Toy{PC:100, N:Play-Doh, H:101, R:true, W:5.050000000000001}, PlayDough{C:GREEN}
  • specialPlay() Each toy has its own defined behavior when being played with and is therefore abstract here.
  • toString(): Returns a string representation of the toy in the format (must be exact): 
                Toy{PC:?, N:?, H:?, R:?, W:?}
    • PC for product code
    • N for name
    • H for current happiness
    • R for retired (true or false)
    • W for current wear
    For example: 
                Toy{PC:100, N:Play-Doh, H:40, R:false, W:2.0}

PlayDough

PlayDough is a concrete class that inherits from Toy. It has the following behaviors:
  • A constructor that takes the name and color (specified with the Color enum). The unique product code starts at 100 and increases by 1 each time a new one is created.
  • getColor(): Returns the color.
  • specialPlay(): The following things happen in order:
    1. A tab indented, "\t", message is displayed to standard output in the format: 
                      Arts and crafting with {color} {name}
      For example: 
                      Arts and crafting with GREEN Play-Doh
    2. The wear increases by a multiple of WEAR_MULTIPLIER * time.
  • toString(): Returns a string representation in the format: 
                Toy{PC:?, N:?, H:?, R:?, W:?}, PlayDough{C:?}
    • The Toy part of the string comes from Toy's toString()
    • C for the color
    For example: 
                Toy{PC:100, N:Play-Doh, H:40, R:false, W:2.0}, PlayDough{C:GREEN}

Doll

Doll is a concrete class that inherits from Toy. It has the following behaviors:
  • A constructor that takes the name, hair color, age and spoken catchphrase. The unique product code starts at 200 and increases by 1 each time a new one is created.
  • A constructor used by subclass ActionFigure that takes its product code plus all the same arguments as the first constructor.
  • getHairColor(): Returns the hair color.
  • getAge(): Returns the age.
  • getSpeak(): Returns the catchphrase.
  • specialPlay(): The following things happen in order:
    1. A tab indented, "\t", message is displayed to standard output in the format: 
                          {name} brushes their {hair color} hair and says, "{speak}"
      For example: 
                          GabbyGabby brushes their RED hair and says, "Will_you_be_my_friend?"
    2. The wear increases by the age.
  • toString(): Returns a string representation in the format: 
                Toy{PC:?, N:?, H:?, R:?, W:?}, Doll{HC:?, A:?, S:?}
    • The Toy part of the string comes from Toy's toString()
    • HC for the hair color
    • A for the age
    • S for the speak
    For example: 
                Toy{PC:200, N:GabbyGabby, H:101, R:true, W:18.0}, Doll{HC:RED, A:6, S:Will_you_be_my_friend?}

ActionFigure

ActionFigure is a concrete class that inherits from Doll. It has the following behaviors:
  • A constructor that takes the name, age and spoken catchphrase. All action figures have HAIR_COLOR (orange) hair. The unique product code starts at 300 and increases by 1 each time a new one is created.
  • getEnergyLevel(): Returns the energy level (starts at MIN_ENERGY_LEVEL and increases when played with).
  • specialPlay(): The following things happen in order:
    1. A tab indented, "\t", message is displayed to standard output in the format (where the energy forumla is computed as: energy * time): 
                          {name} kung foo chops with {energy-formula} energy!
      For example: 
                          He-man kung foo chops with 46 energy!
    2. The energy is increased by 1.
    3. The special play of Doll is called.
  • toString(): Returns a string representation in the format: 
                Toy{PC:?, N:?, H:?, R:?, W:?}, Doll{HC:?, A:?, S:?}, ActionFigure{E:?}
    • The Toy part of the string comes from Toy's toString()
    • The Doll part of the string comes from Doll's toString()
    • E for the energy
    For example: 
                Toy{PC:300, N:He-man, H:63, R:false, W:60.0}, Doll{HC:ORANGE, A:30, S:By_the_power_of_Grayskull!}, ActionFigure{E:3}

BatteryPowered

BatteryPowered is an abstract subclass of Toy. RCCar and Robot inherit from this class to obtain its state and behaviors.

  • A constructor that takes the product code, name and number of batteries.
  • getBatteryLevel(): Returns the current battery level (starts at FULLY_CHARGED).
  • getNumBatteries(): Returns the number of batteries.
  • useBatteries(): When played with the batteries are used and the following things happen in order:
    1. The battery level is reduced by: time + number-of-batteries.
    2. If the battery level is depleted, the following happens in order:
      1. The battery level is capped at DEPLETED
      2. A tab indented, "\t", depletion message is displayed in the format: 
        3.                     DEPLETED:toy-toString
      3. The battery level is restored to FULLY_CHARGED.
      4. A tab indented, "\t", recharge message is displayed in the format: 
                            RECHARGED:toy-toString
  • toString(): Returns a string representation in the format: 
                Toy{PC:?, N:?, H:?, R:?, W:?}, BatteryPowered{BL:?, NB:?}
    • The Toy part of the string comes from Toy's toString()
    • BL for the battery level
    • NB for the number of batteries
    For example: 
                Toy{PC:300, N:Herbie, H:101, R:true, W:45.0}, BatteryPowered{BL:100, NB:4}

RCCar

RCCar is a concrete class that inherits from BatteryPowered. It has the following behaviors:
  • A constructor that takes the name and number of batteries. The unique product code starts at 400 and increases by 1 each time a new one is created.
  • getSpeed(): Returns the current speed (starts at STARTING_SPEED and increases when played with).
  • specialPlay(): The following things happen in order:
    1. A tab indented, "\t", message is displayed to standard output in the format: 
                          {name} races around at {speed}mph!
      For example: 
                          Herbie races around at 10mph!
    2. The batteries are used for the amount of time.
    3. The wear increases by the current speed.
    4. The speed increases by SPEED_INCREASE.
  • toString(): Returns a string representation in the format: 
                Toy{PC:?, N:?, H:?, R:?, W:?}, BatteryPowered{BL:?, NB:?}, RCCar{S:?}
    • The Toy part of the string comes from Toy's toString()
    • The BatteryPowered part of the string comes from BatteryPowered's toString()
    • S for the speed
    For example: 
                Toy{PC:400, N:Herbie, H:40, R:false, W:10.0}, BatteryPowered{BL:56, NB:4}, RCCar{S:15}

Robot

Robot is a concrete class that inherits from BatteryPowered. It has the following behaviors:
  • A constructor that takes the name, number of batteries and whether it flies or not. The unique product code starts at 500 and increases by 1 each time a new one is created.
  • isFlying(): Returns whether it flies or not.
  • getDistance(): Returns the total distance traveled when played with (starts at INITIAL_SPEED).
  • specialPlay(): The following things happen in order:
    1. If flying:
      1. The distance is increased by time * FLY_SPEED
      2. A tab indented, "\t", message is displayed to standard output in the format: 
                            {name} is flying around with total distance: {distance}
        For example: 
                            Voltron is flying around with total distance: 2675
      3. The wear increases by FLY_SPEED
    2. If not flying:
      1. The distance is increased by time * RUN_SPEED
      2. A tab indented, "\t", message is displayed to standard output in the format: 
                            {name} is running around with total distance: {distance}
        For example: 
                            Wall-E is running around with total distance: 700
      3. The wear increases by RUN_SPEED
    3. The batteries are used for the amount of time.
  • toString(): Returns a string representation in the format: 
                Toy{PC:?, N:?, H:?, R:?, W:?}, BatteryPowered{BL:?, NB:?},  Robot{F:?, D:?}
    • The Toy part of the string comes from Toy's toString()
    • The BatteryPowered part of the string comes from BatteryPowered's toString()
    • F for flying or not (true or false)
    • D for distance
    For example: 
                Toy{PC:503, N:IronGiant, H:73, R:false, W:50.0}, BatteryPowered{BL:3, NB:12}, Robot{F:true, D:1825}

In-Lab Activity

The goal of the In-Lab activity is to begin developing the toy hierarchy. You are required to fully implement:

  • IToy: The public Toy interface used by the JUnit tests, the toy factory, and the main program.
  • Toy: The abstract superclass of the entire toy hierarchy.
  • PlayDough: A concrete toy that can be made and played with.

In addition, you will be required to stub out the remaining classes in the Toy hierarchy so that all the JUnit tests run.

Stubbing means we only care about providing the correct signatures of the public methods the tester uses. We do not care about making this class function correctly yet. That will come later once we can compile cleanly. Do not worry about adding state to these classes yet. Using the provided UML class diagram in the design section above, stub out all the concrete toy classes.

  • Constructors and methods that do not return anything do not need any code in their body and can remain empty.
  • Methods that return something need a single statement that returns something in order to compile. It is suggested you return the following based on the return type:
    • For integers and doubles return 0
    • For booleans return false
    • For objects, e.g. String, return null

Your final project structure should look like the following when finished:

Only TestPlayDough is required to run correctly and it is expected to pass all tests.

Keep in mind that only the public behaviors and constants are being provided you in the Design. You must provide the private state on your own.

Also one final note about the JUnit tests. Because of the statics that are used for the product codes, you should not run the entire suite of tests together. Only run each test individually or else you will see some differences.

Design Reminder

The UML class diagram gives you direct hints about where to put your private state. The presence of an accessor and/or mutator means the private state must go in that class. It is very important that you do not replicate state in subclasses! For example, the name of each toy belongs in the Toy class. If it needs to be accessed (or modified) in a subclass, the subclass should use the appropriate superclass accessor/mutator.

The use of instanceof anywhere in this assignment is strictly prohibited! The natural polymorphic behavior of the toys must be adhered to throughout the entire implementation.

Lab Activity

The full lab implementation is broken into two parts:

  • Finish implementing the remaining toys in the hierarchy.
  • Complete the main program that runs the simulation.

Completing the Toy Hierarchy

We suggest you follow this order for completing the toy hierarchy. As you develop each new class make sure to verify it with the corresponding JUnit test before moving on to the next class.

  1. Doll: Tested by TestDoll
  2. ActionFigure: Tested by TestActionFigure
  3. BatteryPowered: Abstract superclass inherited by RCCar and Robot
  4. RCCar: Tested by TestRCCar
  5. Robot: Tested by TestRobot

Completing the ToyLand

The next activity is finishing the implementation of the main program, Toyland, so the toys can be created and played with.

The data directory contains various data files representing the various toys used in the simulation.

The output directory contains the expected outputs for each data file.

As mentioned earlier, there are run configurations provided to you that run with each of the toy data files.

Look over the code provided for Toyland (in the toyland package). It already does the work of reading the toy file from the command line and creating them. You need to implement the playTime() and displayStatistics() methods.

Be very careful here! You should only be using IToy in your implementation! The main program is only aware of this interface. It is intentionally unfamiliar with any of the actual toys in the hierarchy.

Creating Your Own Toy!

For the final part of the lab, you will create your very own type of toy. The only requirement is that it fits somewhere into the existing toy hierarchy you just finished developing. Also it goes without saying but please make your own toy appropriate and non-offensive.

  • Decide what type of toy you want to create. Think about what kind of state and special play behavior it will have.
  • Create your new toy by completely implementing it.
  • In order to test your toy, you will need to make a data file for it. Please use the provided mytoy-5.txt file in the data directory to add your data in. You will specify five different instances of this toy. Take a look at the other input files to get an idea of the format. The first string must be the name of your toy, followed by its properties.
  • In order to test your new toy, you should use the provided run configuration Toyland mytoy-5.txt. When you first run this you will run into problems because all the toys are null. This is because the toy factory does not initially recognize your new toy.
  • To fix this you need to make two additions to ToyFactory in the toys package.
    • Add your new toy type to the end of the ToyType enum. The name of the enum should match the first strings in the data file. You should use all capitalization.
    • In the factory method, makeToy, you should add a case to deal with creating your new toy when a line is encountered that specifies it.
  • When finished, you should be able to create five of your toys, play with them until they are retired, and then display statistics about them.
